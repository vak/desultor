# Future orchestrator replacement

## Problem

Desultor is deliberately a stopgap. It improves project hygiene, but it does
not solve orchestration as a first-class systems problem.

## What a serious replacement should do better

- keep explicit state across harnesses without relying on ad hoc chat residue;
- manage specialist roles without duplicating work;
- make review, handoff, and ownership transfer first-class;
- keep provenance and durable project memory without manual repetition;
- reduce the amount of procedural markdown that humans and harnesses must carry.

## Migration constraint

Any future replacement should preserve:

- explicit durable project artifacts;
- honest provenance and review trails;
- clean separation between stable contracts, findings, and open design work.

## Open question

Should the future system primarily replace:

- the helper scripts;
- the lifecycle markdown conventions;
- the harness-instruction files;
- or all of them through a single governing runtime?

This remains open on purpose.
